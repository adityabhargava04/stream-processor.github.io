Infrastructure

The repository, build system, and CUDA environment are configured. We set up an OpenGL rendering pipeline using GLAD and GLFW directly, without additional abstraction libraries.

Particle System

We implemented the particle system and integrated it with the basic rendering pipeline, allowing particles to be initialized and displayed as point sprites. Particle data is currently stored as a struct-of-arrays layout in anticipation of GPU memory access patterns, where coalesced reads across threads require the same field for many particles to be contiguous in memory.

MAC Grid & P2G Transfer (CPU Reference)

On the CPU side, we implemented the two foundational components of the PIC/FLIP pipeline: the MAC (Marker-And-Cell) grid data structure, which stores velocity components on cell faces and pressure at cell centres, and the particle-to-grid (P2G) velocity transfer, which splats each particle's velocity onto nearby grid nodes using trilinear interpolation weights. These serve as a reference implementation ahead of GPU parallelisation.

One significant design decision: rather than a mixed CPU/GPU pipeline, we are committing to a fully GPU-resident simulation. The particle update is straightforwardly parallelisable, and keeping all simulation state on the GPU avoids costly CPU-GPU transfers each frame. The main technical challenge is the P2G transfer, which introduces a race condition when multiple particle threads write to the same grid cell. We are evaluating two approaches: atomic operations (simpler, but may bottleneck under high particle density) and a sort-based scheme where particles are hashed by cell ID, sorted, and processed by a subsequent per-cell kernel pass.
